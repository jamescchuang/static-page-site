推理模型 vs 非推理模型

根據 2025 年 10 月的最新數據，OpenAI 的「推理模型」（Reasoning Models，如 GPT-5 系列）展現了顯著的進步。這些模型在生成回答前會進行內部的「思考」過程，這似乎起到了類似自我代碼審查的作用。

相比之下，非推理模型的表現大多停滯在 50%-59% 的區間，沒有顯著提升。除了 OpenAI 之外，其他供應商（如 Anthropic, Google, Meta）的新模型並未在基準測試中顯示出安全性的重大突破。

為什麼 Java 表現這麼差？

數據顯示 Java 的安全性合格率僅有 28.5%，遠低於 Python (61.69%) 或 JavaScript (57.34%)。報告推測這與訓練數據的本質有關。

報告見解： Java 擁有悠久的伺服器端開發歷史，許多訓練數據來自於 SQL Injection 等漏洞被廣泛認知之前的舊程式碼。因此，模型可能從含有大量歷史漏洞的數據集中「學壞了」。

漏洞類型的極端差異

模型在處理不同類型的安全漏洞時表現出極大的差異：

  • 表現良好：SQL Injection (SQLi) 和加密演算法 (Crypto) 的合格率超過 80%。這是因為這類問題通常有標準的「正確答案」（例如使用 Parameterized Queries）。
  • 表現極差：跨站腳本攻擊 (XSS) 和 Log Injection 的合格率僅約 12-13%。這類問題需要理解上下文（Context）來判斷變數是否被污染，這對 LLM 來說極具挑戰性。

結論與建議

儘管新型推理模型帶來了希望，但目前 AI 生成的程式碼中仍有近一半包含安全風險。單靠 AI 生成安全程式碼在現階段是不可行的。

企業應採取的措施：

  • 繼續使用 SAST (靜態應用程式安全測試) 和 SCA 工具。
  • 實施嚴格的人工代碼審查。
  • 不要假設 AI 生成的代碼是安全的，必須將其視為不可信的輸入。
