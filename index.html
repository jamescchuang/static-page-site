AI 正在徹底改變軟體開發

開發人員正從直接編寫程式碼，轉向以自然語言描述功能，並利用大型語言模型 (LLM) 生成具體程式碼。

然而，開發者通常不會在提示中明確指定安全約束。選擇權落在了模型身上。

AI 生成的程式碼包含已知安全漏洞

趨勢一：語法完美，安全停滯

● 語法正確率 趨近完美，近一年的模型幾乎都能生成可編譯的程式碼。

● 安全效能 低迷且穩定，新模型僅比舊模型略有改善，整體趨勢持平。

Java 的表現令人擔憂

Python (61%), C# (57%), JS (55%) 的表現非常相似且穩定。

異常值：Java 的表現明顯較差，平均通過率僅為 28.50%。

原因：歷史遺留問題。Java 訓練資料中可能包含了更多在 SQL 注入被廣泛認識之前編寫的不安全代碼。

漏洞類型的極端差異

  • ✅
    表現良好：
    不安全加密 (85%) & SQL 注入 (80%)
  • ❌
    表現極差：
    跨網站指令碼 (13%) & 日誌注入 (12%)

原因：SQLi 有標準的「預備陳述式」修復方案。而 XSS 需要複雜的上下文分析，模型難以判斷變數是否受使用者控制。

深入探討：根本性挑戰

為什麼模型規模越大不代表安全性越高？
結果顯示，模型大小對安全效能的影響微乎其微。大型 (>100B 參數) 通過率為 50.87%，與中型和小型的表現幾乎一致。這表明單純增加參數無法解決根本的安全認知問題。
LLM 與靜態分析引擎 (SAST) 的區別？
  • SAST： 執行精確的、跨程序的數據流分析，追蹤數據源頭。
  • LLM： 依賴有限的上下文視窗 (Context Window) 和淺層線索 (如變數名稱) 進行猜測，難以執行深度語意計算來判斷數據是否「髒」。
為什麼安全性沒有隨著模型能力提升而改善？
這是訓練資料本質的問題。網路上的程式碼範例大多語法正確，但其中許多包含未修復的安全漏洞。模型學習了安全和不安全的實作方式，並認為兩者都是「合法」的。
❝

雖然大型語言模型已擅長生成功能正確的程式碼，
但它們仍然以驚人的高頻率引入安全漏洞。

AI 不會主動學習安全

模型的進步集中在功能性，安全性的鴻溝依然存在。

單純依賴 AI 是危險的

在沒有專門安全工具輔助下，將代碼整合到生產環境風險巨大。

開發者需要意識與工具

必須配備能夠在開發流程中檢測和修復這些漏洞的工具。
